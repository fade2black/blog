    Kyber + Dilithium AKE Protocol Specification

    Status: Draft — Phase 1 (Specification only)

    1. Roles

    • Client — initiates handshake, derives session key.
    • Server — responds, authenticates using Dilithium signature.

    2. Cryptographic Primitives

    Purpose Algorithm
    Key Encapsulation (KEM) Kyber (ephemeral per session)
    Signatures Dilithium (long-term server key)
    Hash / Transcript SHA-256
    Key Derivation HKDF-SHA256
    AEAD (optional for Phase 2) ChaCha20-Poly1305 or AES-GCM

    3. Handshake Overview

    Goal: derive a shared session key K between client and server.

    Flow (2.5 round trips)

      sequenceDiagram
    Client-->>Server: ClientHello
    Server-->>Client: ServerHello
    Client-->>Server: ClientKEM

    3.1 ClientHello

    Field Length Notes
    client_id_len u8 length of client_id
    client_id variable optional, UTF-8
    client_nonce 32 bytes random

    Rust struct: 

    pub struct ClientHello {
    pub client_id: Vec<u8>,       // max 255 bytes
    pub client_nonce: [u8; 32],
}

    3.2 ServerHello

    Field Length Notes
    server_id_len u8 length of server_id
    server_id variable UTF-8
    server_nonce 32 bytes random
    pk_kyber \~1184 bytes Kyber768 public key
    sig_dilithium \~2701 bytes Dilithium2 signature

    The server computes

    sig = Sign(sk_dil, pk_kyber || client_nonce || server_nonce || server_id)

    3.3 ClientKEM

    Field Length Notes
    ciphertext \~1088 bytes Kyber768 ciphertext

    4. Transcript Binding

    • Maintain a running SHA-256 hash of all handshake messages (in serialized order).
    • Server signs pk_kyber || client_nonce || server_nonce || server_id to bind its ephemeral key to the session.
    • After ClientKEM, both sides compute: 
      transcript_hash = SHA256(ClientHello || ServerHello || ClientKEM)
session_key = HKDF-Expand(ss, transcript_hash || "Kyber-Dilithium-AKE")
    • Server zeroizes sk_kyber after decapsulation.

    5. Encoding Rules

    • Variable-length fields are prefixed with u8 length.
    • Fixed-length fields are written raw.
    • This gives a deterministic canonical encoding for hashing/signing.

    6. Future Extensions (Phase 2+)

    • Client authentication using Dilithium
    • Session resumption via PSK
    • AEAD-encrypted application channel using session_key

    7. Error Handling Rules

    We want to clearly define what should happen if something goes wrong during the handshake.

    Error Type When it occurs Action / Outcome
    Malformed message Parsing fails (e.g., wrong length prefix, truncated data) Abort handshake, log error
    Invalid signature Dilithium signature verification fails Abort handshake, do not derive keys
    Decapsulation failure Kyber decapsulation fails (invalid ciphertext) Abort handshake, do not derive keys
    Replay detected client_nonce or server_nonce seen before in short-term cache Abort handshake
    Unexpected message Message arrives out of expected order Abort handshake, log warning
    Crypto library failure Any KEM/signing/hash function fails Abort handshake, log error

    Notes:

    • Always zeroize ephemeral secrets (sk_kyber, ss) immediately on abort.
    • Log errors for debugging but never leak sensitive info (don’t include secrets in logs).
    • Clients may retry with a new ClientHello if appropriate.

    Handshake State Machine

    We can represent the handshake as a finite state machine (FSM) with explicit states.

    Client FSM

    Current State Event/Action Next State
    Init send_client_hello() SentClientHello
    SentClientHello receive_server_hello() + verify sig ReceivedServerHello
    ReceivedServerHello send_client_kem_ciphertext() SentKemCiphertext
    SentKemCiphertext finalize() (derive session keys) Complete
    any error (bad sig, malformed msg, etc.) Failed 
    stateDiagram-v2
    [*] --> Init
    Init --> SentClientHello: send_client_hello()
    SentClientHello --> ReceivedServerHello: receive_server_hello() + verify
    ReceivedServerHello --> SentKemCiphertext: send_client_kem()
    SentKemCiphertext --> Complete: finalize()
    [*] --> Failed: error

    Server FSM

    Current State Event/Action Next State
    Init receive_client_hello() ReceivedClientHello
    ReceivedClientHello send_server_hello() SentServerHello
    SentServerHello receive_client_kem_ciphertext() (derive session keys) Complete
    any error (bad msg, malformed data, etc.) Failed 
    stateDiagram-v2
    [*] --> Init
    Init --> ReceivedClientHello : receive_client_hello()
    ReceivedClientHello --> SentServerHello : send_server_hello()
    SentServerHello --> Complete : receive_client_kem_ciphertext()
    [*] --> Failed : error

    Handshake Interaction

    sequenceDiagram
    autonumber
    participant Client
    participant Server

    Note over Client,Server: Setup
    Client->>Server: TCP connect
    Server-->>Client: accept connection

    Note over Client,Server: Handshake Step 1
    Client->>Server: send_client_hello()
    Server->>Server: receive_client_hello()
    Server-->>Client: acknowledgment implicitly via next step

    Note over Client,Server: Handshake Step 2
    Server->>Client: send_server_hello()
    Client->>Client: receive_server_hello() + verify signature with server public key

    Note over Client,Server: Handshake Step 3
    Client->>Server: send_client_kem()
    Server->>Server: receive_client_kem() + derive shared secret

    Note over Client,Server: Handshake Complete
    Client->>Client: finalize() + derive shared secret
    Server->>Server: handshake complete, session keys established
